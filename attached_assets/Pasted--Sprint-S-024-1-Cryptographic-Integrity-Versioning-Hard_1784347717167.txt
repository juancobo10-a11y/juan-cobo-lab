# Sprint S-024.1 — Cryptographic Integrity & Versioning Hardening

## Contexto

El Sprint S-024 incorporó a HELIOS:

* snapshots inmutables;
* versionado lineal;
* comparación entre versiones;
* paquetes `.helios.json`;
* importación y exportación;
* migraciones de esquema;
* reconstrucción de informes;
* interfaces de gestión de versiones.

La implementación quedó funcional y con regresión completa en verde. Sin embargo, la revisión arquitectónica identificó una desviación crítica y varias áreas de endurecimiento:

1. Se utilizó `MurmurHash3-128`, que no es un hash criptográfico.
2. No está suficientemente demostrado que el diff distinga `modified` de `reordered`.
3. La canonización determinística requiere mayor cobertura.
4. La importación debe ser estrictamente transaccional.
5. El registro de migraciones no debe depender de efectos secundarios en la UI.
6. La inmutabilidad profunda debe demostrarse explícitamente.
7. La documentación debe diferenciar integridad, autenticidad y firma digital.
8. La suite S-024 quedó por debajo de la cobertura solicitada.

Este sprint es exclusivamente de hardening.

No agregar nuevas capacidades funcionales.

No agregar nuevas pantallas salvo ajustes mínimos de mensajes o estados.

No modificar el modelo metodológico.

No modificar el Thinking Router.

No introducir IA generativa, LLM, APIs externas ni firma digital.

No hacer republish.

---

# Objetivo general

Cerrar técnicamente S-024 mediante:

```text
Hash no criptográfico
↓
SHA-256
↓
Canonización determinística
↓
Importación transaccional
↓
Migraciones declarativas
↓
Diff semántico robusto
↓
Reproducibilidad verificable
```

Al finalizar, HELIOS debe ofrecer integridad criptográfica interna, sin afirmar autenticidad ni autoría.

---

# ADR-0013A

Agregar:

## ADR-0013A — La integridad de snapshots y paquetes requiere hash criptográfico

MurmurHash3 no debe utilizarse para verificar integridad frente a manipulación deliberada.

HELIOS debe usar SHA-256 para:

* snapshots;
* paquetes;
* reportes reproducibles;
* verificaciones de integridad.

La integridad criptográfica no acredita:

* autoría;
* identidad;
* origen;
* firma;
* autenticidad institucional.

Una firma digital será una capacidad futura distinta.

# 2. Reemplazo de MurmurHash3 por SHA-256

Sustituir completamente MurmurHash3 en:

```ts
computeSnapshotHash()
computePackageHash()
computeReportHash()
verifySnapshotIntegrity()
verifyProjectPackageIntegrity()
verifyReportReproducibility()
```

No mantener MurmurHash como fallback de integridad.

Eliminar código muerto, dependencias, helpers y documentación asociada.

---

# 3. Estrategia SHA-256

Preferir:

```ts
crypto.subtle.digest("SHA-256", data)
```

Siempre que el entorno de navegador y tests lo soporte.

Si se utiliza Web Crypto, convertir las APIs afectadas a `async` de manera explícita.

Ejemplo:

```ts
async function computeSnapshotHash(...): Promise<string>
```

No ocultar asincronía mediante estados globales o callbacks improvisados.

Si el entorno de tests no expone `crypto.subtle`, usar un adaptador controlado:

```ts
CryptoHashAdapter
```

con implementación:

* navegador: Web Crypto;
* tests Node: `node:crypto`.

No implementar SHA-256 manualmente salvo imposibilidad técnica documentada.

---

# 4. CryptoHashAdapter

Crear:

```text
src/project-versioning/crypto/CryptoHashAdapter.ts
```

Interfaz:

```ts
export interface CryptoHashAdapter {
  sha256(input: string | Uint8Array): Promise<string>;
}
```

Implementaciones:

```ts
WebCryptoHashAdapter
NodeCryptoHashAdapter
```

El servicio de dominio debe depender de la interfaz o de una función inyectable, no de `window`.

No duplicar lógica de conversión hexadecimal.

# 5. Canonización determinística

Crear o endurecer:

```ts
canonicalize()
canonicalStringify()
normalizeSnapshotPayload()
normalizeProjectPackage()
normalizeGeneratedReport()
```

La salida canónica debe:

* ordenar claves de objetos;
* preservar `null`;
* excluir `undefined`;
* rechazar referencias circulares;
* representar fechas como strings normalizados;
* conservar tipos JSON válidos;
* manejar Unicode consistentemente;
* no depender del orden accidental de creación de propiedades.

```ts
value.normalize("NFC")
```

Aplicar de forma consistente antes del hash.

Agregar pruebas con:

* caracteres acentuados;
* caracteres compuestos y descompuestos;
* emojis;
* caracteres especiales.

Documentar la decisión.

# 7. Orden semántico

Clasificar expresamente las colecciones.

## Preservar orden

Cuando el orden representa una decisión del usuario:

* variables;
* indicadores;
* filas de operacionalización;
* filas de contrastación;
* evidencias;
* secciones del reporte;
* hipótesis ordenadas editorialmente;
* versiones lineales cuando corresponda.

## Ordenar establemente

Cuando el orden no tiene significado:

* tags;
* sets conceptuales;
* listas de IDs de referencias principales, si la UI no les asigna jerarquía;
* claves de mapas;
* catálogos declarativos.

Documentar cada decisión.

No ordenar todos los arrays indiscriminadamente.

---

# 8. Hash de snapshot

El hash debe calcularse sobre:

```text
schemaVersion
+
metadata estable relevante
+
payload normalizado
```

Excluir:

* `createdAt`;
* `contentHash`;
* estado UI;
* auditorías derivadas;
* previews;
* reportes generados;
* timestamps de ejecución.

Definir explícitamente qué campos de metadata entran al hash.

Recomendación:

Incluir:

* `projectId`;
* `projectName`;
* `versionLabel`;
* `description`;
* `author`;
* `tags`;
* `sourceSnapshotId`;
* `changeSummary`.

Excluir únicamente campos operativos temporales.

---

# 9. Hash de paquete

El hash del paquete debe calcularse sobre:

* manifest estable;
* snapshots normalizados;
* versiones normalizadas.

Excluir:

* `packageHash`;
* `exportedAt`, si la reproducibilidad binaria lo requiere.

La decisión sobre `exportedAt` debe documentarse y probarse.

No afirmar autenticidad del paquete.

---

# 10. Hash de reporte reproducible

Crear o endurecer:

```ts
computeGeneratedReportHash()
```

Excluir:

* fecha de ejecución;
* timestamps de exportación;
* metadatos transitorios.

Incluir:

* estructura;
* bloques;
* contenido;
* trazabilidad;
* orden de secciones.

El mismo snapshot y `ReportDefinition` deben producir el mismo hash.

# 11. Compatibilidad de APIs

Mantener compatibilidad razonable con S-024.

Si las funciones deben convertirse en asíncronas:

* actualizar consumidores;
* actualizar UI;
* actualizar tests;
* actualizar documentación;
* evitar wrappers engañosamente síncronos.

No mantener dos implementaciones paralelas de hash.

---

# 12. Inmutabilidad profunda

Endurecer:

```ts
createProjectSnapshot()
cloneSnapshot()
reconstructSessionFromSnapshot()
```

Garantizar:

* deep copy;
* sin referencias compartidas;
* snapshot no afectado por mutaciones posteriores;
* sesión reconstruida independiente del snapshot;
* una reconstrucción no muta el original.

Aplicar `deepFreeze()` al snapshot en entornos compatibles.

Crear:

```ts
deepClone()
deepFreeze()
```

como utilidades puras y probadas.

No depender únicamente de `Readonly<T>`, porque es garantía de TypeScript, no de runtime.

---

# 13. Diff semántico

Endurecer `VersionComparisonService`.

Debe distinguir:

```text
added
removed
modified
reordered
unchanged
```

Comparar por ID, nunca solo por índice.

---

# 14. Reordered

Definir `reordered` como:

```text
misma entidad
+
mismo contenido canónico
+
posición distinta
```

Ejemplos:

* misma evidencia, otra posición;
* misma sección de informe, otro orden;
* misma variable, otra posición.

No clasificar como `modified` si el contenido no cambió.

---

# 15. Modificado y reordenado simultáneamente

Cuando una entidad cambia contenido y posición, definir una regla explícita.

Recomendación:

```text
changeType = modified
reordered = true
```

Extender `ProjectEntityChange` con:

```ts
reordered?: boolean;
beforeIndex?: number;
afterIndex?: number;
```

No perder información de reordenamiento.

Documentar la precedencia.

---

# 16. Breaking changes

Formalizar un catálogo:

```text
src/project-versioning/breaking-change-rules.ts
```

Reglas mínimas:

* hipótesis eliminada;
* variable eliminada con dependientes;
* indicador eliminado y usado;
* fuente eliminada y usada;
* criterio eliminado con evidencia;
* evidencia eliminada y usada en conclusión;
* referencias rotas;
* schema incompatible.

No clasificar como breaking:

* corrección ortográfica;
* edición de descripción;
* reordenamiento;
* actualización de notas;
* cambio de tags.

Cada breaking change debe indicar la regla aplicada.

# 19. Commit atómico

Crear:

```ts
commitImportTransaction()
```

Debe:

* recibir el estado activo;
* recibir una transacción `ready`;
* producir un nuevo estado;
* no mutar el original;
* fallar completamente si alguna validación final falla.

No realizar múltiples `setState` parciales durante la importación.

---

# 20. Estrategias

## create-copy

Debe:

* crear nuevo `projectId`;
* preservar relaciones internas;
* evitar colisiones;
* producir una copia independiente.

Definir si los IDs internos también se regeneran.

Recomendación:

* regenerar `projectId`;
* conservar IDs internos si no hay conflicto;
* remapear IDs internos si hay conflicto;
* mantener tabla explícita de remapeo.

---

## replace-current

Debe:

* requerir confirmación explícita;
* reemplazar en una única operación;
* preservar un snapshot de seguridad opcional solo si ya existe infraestructura para ello;
* no ejecutarse por selección accidental.

---

## cancel

Debe:

* no mutar estado;
* limpiar estado temporal;
* conservar el archivo original intacto.

---

# 21. Validación de paquetes internos

No basta con validar el hash del manifest.

Verificar:

* hash de cada snapshot;
* referencias de versiones a snapshots;
* parentVersionId válido;
* projectId consistente;
* schemaVersion consistente;
* IDs duplicados;
* referencias internas rotas;
* reportDefinitions existentes;
* ausencia de ciclos en versiones.

# Criterio de éxito

S-024 solo podrá considerarse plenamente cerrado cuando HELIOS demuestre que:

```text
Mismo contenido
↓
misma representación canónica
↓
mismo SHA-256
```

y que:

```text
Importación
↓
validación completa
↓
preview
↓
confirmación explícita
↓
commit atómico
```

Además, debe distinguir correctamente:

```text
integridad
≠
autenticidad
≠
firma digital
```

y:

```text
modified
≠
reordered
```

Al finalizar S-024.1, HELIOS deberá poder afirmar únicamente:

> El sistema verifica mediante SHA-256 que el contenido coincide con su representación canónica almacenada.

No deberá afirmar que el archivo fue creado por una persona determinada, que su origen está certificado o que cuenta con firma digital.
