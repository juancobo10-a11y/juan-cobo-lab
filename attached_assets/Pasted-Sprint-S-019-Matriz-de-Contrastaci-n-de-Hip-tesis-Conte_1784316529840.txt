La siguiente etapa consiste en definir explícitamente cómo una hipótesis podría ser contrastada con evidencia.

Este sprint implementa una Matriz de Contrastación, cuyo propósito es organizar de manera estructurada la lógica de validación de cada hipótesis.

No agregar nuevos Thinking Patterns.

No modificar el Router.

No modificar el Hypothesis Builder.

No modificar el Modelo Conceptual.

No modificar la Matriz de Operacionalización.

No introducir IA generativa, LLM, APIs externas ni bases de datos.

Objetivo general

Agregar una nueva etapa:

Hipótesis
↓
Modelo Conceptual
↓
Matriz de Operacionalización
↓
Matriz de Contrastación

La matriz permitirá responder:

¿Qué evidencia apoyaría esta hipótesis?
¿Qué evidencia la debilitaría?
¿Cómo podría observarse?
¿Con qué indicadores?
¿Con qué fuentes?
¿Qué limitaciones existen?
ADR-0008

Agregar:

ADR-0008 — Toda hipótesis debe tener criterios explícitos de contrastación

Una hipótesis no está lista para ser utilizada mientras no existan criterios claros para decidir qué tipo de evidencia la apoyaría o la refutaría.

HELIOS no debe decidir si una hipótesis es verdadera.

Debe ayudar al usuario a definir cómo podría contrastarse.

3. Prioridad

media

No asignar automáticamente.

4. Evidencia esperada

Texto libre.

Debe responder:

¿Qué observaríamos si la hipótesis fuera consistente?

5. Evidencia contraria

Texto libre.

Debe responder:

¿Qué resultado pondría seriamente en duda la hipótesis?

6. Criterio de interpretación

Texto libre.

Ejemplo:

Si el indicador permanece estable pese a la intervención, la hipótesis pierde plausibilidad.

No generar automáticamente.
