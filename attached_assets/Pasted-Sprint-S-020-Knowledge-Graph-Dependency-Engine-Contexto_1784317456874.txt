Sprint S-020 — Knowledge Graph & Dependency Engine
Contexto

Después de los Sprint S-011 a S-019, HELIOS implementa un flujo metodológico completo:

↓

↓
Matriz de Contrastación
↓
Revisión metodológica

Sin embargo, todas estas entidades siguen organizadas principalmente como una secuencia lineal.

La siguiente evolución consiste en representar explícitamente las relaciones entre ellas mediante un Knowledge Graph interno.

El objetivo no es agregar una base de datos de grafos.

El objetivo no es agregar visualizaciones complejas.

El objetivo es introducir un modelo interno de nodos y relaciones que permita razonar sobre dependencias, reutilización e impacto de cambios.

No modificar el Thinking Router.

No modificar los Knowledge Packs.

No introducir IA generativa, LLM ni servicios externos.

No cambiar el comportamiento visible del usuario salvo donde sea estrictamente necesario para exponer dependencias.

Objetivo general

Crear un motor interno de relaciones:

Problema
│
├── Hipótesis
│     │
│     ├── Variables
│     │      │
│     │      ├── Indicadores
│     │      │      │
│     │      │      └── Fuentes
│     │      │
│     │      └── Operacionalización
│     │
│     └── Contrastación

Las relaciones deberán ser reutilizables y consultables.

ADR-0009

Agregar:

ADR-0009 — Las relaciones son entidades de primer nivel

HELIOS no debe depender únicamente de jerarquías de objetos.

Las relaciones entre entidades constituyen conocimiento y deben representarse explícitamente.
