Sprint S-021 — Motor de Consistencia Metodológica
Contexto

Después de los Sprint S-011 a S-020, HELIOS ya dispone de una arquitectura metodológica completa:

Problema
↓
Knowledge Pack
↓
Thinking Pattern
↓
Reflexión
↓
Hipótesis
↓
Modelo Conceptual
↓
Variables
↓
Indicadores
↓
Fuentes
↓
Matriz de Operacionalización
↓
Matriz de Contrastación
↓
Revisión metodológica
↓
Knowledge Graph

El Sprint S-020 incorporó un KnowledgeGraphService capaz de reconstruir relaciones, detectar dependencias, calcular impactos, identificar huérfanos y validar la integridad estructural del grafo.

Sin embargo, la integridad estructural no garantiza consistencia metodológica.

Una cadena puede estar técnicamente conectada y, aun así, presentar problemas como:

hipótesis causales sin variable de resultado;
variables sin indicadores;
indicadores vinculados a una hipótesis diferente;
fuentes no utilizadas;
dimensiones operacionalizadas pero no contrastadas;
criterios de contrastación desconectados del modelo conceptual;
relaciones redundantes;
cadenas incompletas.

Este sprint debe crear un motor de auditoría metodológica explicable, construido sobre el estado de sesión y el Knowledge Graph.

No agregar nuevos Thinking Patterns.

No modificar el Thinking Router.

No modificar los Knowledge Packs.

No modificar el comportamiento de los servicios S-015 a S-020, salvo ajustes estrictamente necesarios para integración.

No introducir IA generativa, LLM, APIs externas ni bases de datos.

No generar contenido metodológico automáticamente.

Objetivo general

Crear un motor capaz de evaluar la coherencia entre:

Hipótesis
↓
Variables
↓
Indicadores
↓
Fuentes
↓
Operacionalización
↓
Contrastación

El motor debe detectar inconsistencias, explicarlas y señalar las entidades involucradas.

HELIOS no debe decidir si un diseño es científicamente verdadero, válido o exitoso.

Debe identificar reglas metodológicas incumplidas o potencialmente problemáticas.

ADR-0010

Agregar:

ADR-0010 — La consistencia metodológica se evalúa mediante reglas explícitas y trazables

Las evaluaciones metodológicas no deben depender de heurísticas opacas.

Cada observación debe derivarse de una regla identificable, documentada y verificable.

Toda observación debe incluir:

regla aplicada;
severidad;
explicación;
entidades involucradas;
evidencia estructural;
acción sugerida.

El sistema no debe corregir automáticamente el diseño del usuario.

Categorías de reglas

Crear un catálogo extensible:

type MethodologicalRuleCategory =
  | "hypothesis-structure"
  | "variable-coverage"
  | "indicator-alignment"
  | "evidence-coverage"

Severidad
type MethodologicalSeverity =
  | "error"
  | "warning"
  | "suggestion";

Semántica:

Error

Existe una inconsistencia que impide considerar completa la cadena metodológica.

Warning

Existe una debilidad que no necesariamente invalida el diseño, pero requiere revisión.

Suggestion

Existe una oportunidad de mejora metodológica.

No usar puntuaciones numéricas arbitrarias.

6. Reglas metodológicas mínimas

Implementar como mínimo las siguientes reglas.

A. Estructura de hipótesis
HYP-001 — Hipótesis causal sin variable causal

Aplicar cuando la hipótesis tenga estructura causal.

Debe existir al menos una variable con rol causal o equivalente según el modelo existente.

Severidad: error.

HYP-002 — Hipótesis causal sin variable de resultado

Debe existir al menos una variable con rol de resultado.

Severidad: error.

HYP-003 — Hipótesis sin modelo conceptual

Una hipótesis confirmada debe tener un modelo conceptual asociado.

Severidad: error.

HYP-004 — Hipótesis con formulación no operacionalizable

Usar únicamente criterios estructurales disponibles.

No analizar semánticamente el lenguaje.

Ejemplo de criterio válido:

hipótesis sin variables;
hipótesis sin indicadores;
hipótesis sin operacionalización.

Severidad: warning.

B. Cobertura de variables
VAR-001 — Variable sin indicador

Toda variable operacionalizada debe tener al menos un indicador.

Severidad: error.

VAR-002 — Variable de resultado no contrastada

Toda variable de resultado debe estar representada por al menos un indicador usado en contrastación.

Severidad: warning.

VAR-003 — Variable conceptual sin definición operacional

No duplicar la validación local de S-017.

Esta regla solo debe aparecer cuando afecte la consistencia transversal.

Severidad: warning.

VAR-004 — Variable aislada

Variable conectada a una hipótesis, pero sin operacionalización ni contrastación.

Severidad: warning.

C. Coherencia de indicadores
IND-001 — Indicador sin variable

Indicador sin relación válida con una variable.

Severidad: error.

IND-002 — Indicador no operacionalizado

Indicador existente, pero no utilizado en la matriz de operacionalización.

Severidad: warning.

IND-003 — Indicador no contrastado

Indicador operacionalizado, pero no utilizado en ninguna fila de contrastación.

Severidad: suggestion.

IND-004 — Indicador cruzado entre hipótesis

Una fila de contrastación no debe utilizar un indicador perteneciente exclusivamente a otra hipótesis.

Severidad: error.

Debe usar relaciones del Knowledge Graph, no comparar únicamente identificadores locales.

IND-005 — Indicador redundante

Detectar relaciones duplicadas o indicadores funcionalmente repetidos únicamente cuando exista evidencia estructural:

mismo indicador;
misma variable;
misma dimensión;
misma fuente;
misma hipótesis.

No inferir similitud semántica por texto.

Severidad: warning.

D. Cobertura de fuentes
SRC-001 — Indicador contrastado sin fuente

No duplicar únicamente la validación de fila.

Evaluar la cadena completa:

Contrastación
↓
Indicador
↓
Fuente

Severidad: error.

SRC-002 — Fuente no utilizada

Distinguir explícitamente:

orphan

de:

unused

Una fuente válida pero no utilizada debe clasificarse como suggestion, no como error.

SRC-003 — Fuente utilizada fuera de la hipótesis

Una fila de contrastación no debe reutilizar una fuente desconectada del indicador o de la operacionalización correspondiente, salvo que la estructura actual permita justificarla.

Severidad: warning.

No asumir incompatibilidad temática.

E. Coherencia de operacionalización
OPR-001 — Dimensión sin indicador

Severidad: error.

OPR-002 — Dimensión sin fuente

Severidad: error.

OPR-003 — Dimensión operacionalizada pero no contrastada

Severidad: warning.

OPR-004 — Múltiples filas redundantes

Detectar únicamente duplicación estructural exacta:

misma variable
+
misma dimensión
+
mismo indicador
+
misma fuente

Severidad: warning.

OPR-005 — Unidad o escala ausente

No bloquear el diseño.

Severidad: suggestion.

F. Coherencia de contrastación
CON-001 — Evidencia esperada sin evidencia contraria

Severidad: error.

CON-002 — Evidencia contraria sin criterio de interpretación

Severidad: warning.

CON-003 — Contrastación sin indicador operacionalizado

Severidad: error.

CON-004 — Contrastación sin fuente operacionalizada

Severidad: warning.

CON-005 — Hipótesis sin criterios de contrastación

Hipótesis confirmada sin matriz confirmada.

Severidad: error.

CON-006 — Criterio duplicado

Duplicación exacta:

mismo indicador
+
misma fuente
+
misma evidencia esperada
+
misma evidencia contraria

Severidad: warning.

G. Trazabilidad y completitud
TRA-001 — Cadena metodológica incompleta

Detectar rupturas en:

Hipótesis
↓
Variable
↓
Indicador
↓
Fuente
↓
Operacionalización
↓
Contrastación

Severidad según punto de ruptura:

relación obligatoria ausente: error;
etapa incompleta: warning.
TRA-002 — Referencia rota

Usar el resultado de validateGraph().

Severidad: error.

TRA-003 — Entidad aislada

Distinguir entidades que requieren relación obligatoria de entidades reutilizables.

Severidad:

obligatoria: warning;
reutilizable y válida: suggestion.
TRA-004 — Dependencia circular ilegal

Usar los ciclos detectados por el Knowledge Graph.

Severidad: error.

7. Reglas y limitaciones metodológicas

El motor no debe evaluar:

calidad estadística del indicador;
validez de constructo;
confiabilidad de una fuente;
representatividad muestral;
causalidad real;
suficiencia empírica;
pertinencia temática;
calidad de la redacción;
veracidad de la hipótesis.

Estas limitaciones deben documentarse.

El motor evalúa únicamente consistencia estructural y metodológica explícitamente modelada.

8. Resultado de auditoría

El resumen debe incluir:

interface MethodologicalAuditSummary {
  totalFindings: number;
  errors: number;
  warnings: number;
  suggestions: number;
  categoriesAffected: MethodologicalRuleCategory[];
  blocking: boolean;
  complete: boolean;
}

Reglas:

blocking = existe al menos un error
complete = no existen errores y todas las etapas obligatorias están confirmadas

No crear una calificación de 0 a 100.

No usar semáforos como sustituto de la explicación.

9. Explicabilidad

Cada finding debe responder:

¿Qué regla se incumplió?
¿Qué se encontró?
¿Qué entidades están involucradas?
¿Por qué importa?
¿Qué puede revisar el usuario?

Ejemplo:

Regla: HYP-002
Problema: La hipótesis causal no tiene variable de resultado.
Evidencia: El modelo contiene dos variables causales y ninguna con rol "resultado".
Impacto: No es posible definir qué cambio esperado debe observarse.
Acción sugerida: Agregar o reclasificar una variable de resultado.

No generar explicaciones mediante LLM.

Usar plantillas determinísticas.

15. Correcciones menores de S-020

Aprovechar el sprint para resolver dos deudas acotadas.

15.1 Recorrido topológico explícito

La convención actual es:

source → target

significa:

source depende de target

Renombrar o complementar la API para eliminar ambigüedad.

Preferencia:

topologicalTraversalDependenciesFirst()
topologicalTraversalDependentsFirst()

Mantener compatibilidad con topologicalTraversal() si ya es usado.

No romper pruebas previas.

15.2 Diferenciar orphan y unused

Actualizar tipos o reportes del grafo para distinguir:

orphan

Entidad con relación obligatoria ausente.

unused

Entidad válida sin utilización actual.

Mantener compatibilidad con la API existente cuando sea posible.

Agregar pruebas de regresión específicas.
