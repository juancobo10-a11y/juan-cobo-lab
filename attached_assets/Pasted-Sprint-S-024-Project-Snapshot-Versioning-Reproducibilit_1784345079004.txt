Sprint S-024 — Project Snapshot, Versioning & Reproducibility Engine
Contexto

Después de los Sprint S-011 a S-023, HELIOS ya permite:

Diseñar
↓
Auditar
↓
Evaluar evidencia
↓
Formular conclusiones
↓
Generar informes trazables

El sistema ya dispone de:

estado metodológico estructurado;
Knowledge Graph reconstruible;
auditoría metodológica;
evaluación de evidencia;
conclusiones por hipótesis;
Report Builder;
exportadores HTML, Markdown y JSON.

Sin embargo, todavía no existe una forma de capturar de manera inmutable el estado completo de un proyecto en un momento determinado.

Esto impide responder con precisión preguntas como:

¿Qué versión del proyecto produjo este informe?
¿Qué cambió entre dos revisiones?
¿Qué hipótesis, variables o evidencias fueron modificadas?
¿Puede reconstruirse exactamente un reporte anterior?
¿El archivo importado conserva su integridad?
¿Es compatible con la versión actual del esquema?

Este sprint debe introducir un motor de versionamiento metodológico, snapshots reproducibles e intercambio de proyectos.

No agregar nuevos Thinking Patterns.

No modificar el Thinking Router.

No introducir IA generativa, LLM ni APIs externas.

No implementar colaboración en tiempo real.

No implementar Git interno.

No persistir diffs manualmente.

No duplicar el estado activo del proyecto como una segunda fuente de verdad.

Objetivo general

Agregar esta capacidad:

Proyecto activo
↓
Snapshot inmutable
↓
Versión identificable
↓
Hash de integridad
↓
Comparación
↓
Reproducción de informe
↓
Exportación / importación

HELIOS debe poder capturar, comparar, verificar, exportar, importar y reconstruir versiones metodológicas completas.

ADR-0013

Agregar:

ADR-0013 — Los snapshots son representaciones inmutables y verificables del estado metodológico

Un snapshot debe representar el estado completo y normalizado del proyecto en un momento determinado.

Debe ser:

inmutable;
determinístico;
verificable;
serializable;
migrable;
reconstruible;
independiente de la interfaz.

El snapshot no debe almacenar:

estado transitorio de UI;
filtros;
pantallas abiertas;
posiciones de scroll;
resultados derivados que puedan reconstruirse;
informes generados como archivos.

Los informes deben poder reconstruirse desde el snapshot y la definición del reporte.

4. ProjectSnapshot

Crear:

interface ProjectSnapshot {
  id: string;
  version: string;
  schemaVersion: string;

  metadata: ProjectSnapshotMetadata;
  payload: ProjectSnapshotPayload;

  contentHash: string;
  createdAt: string;
}
Reglas
Debe ser inmutable.
contentHash debe calcularse sobre el contenido normalizado.
El hash no debe depender de createdAt.
El hash no debe depender del orden accidental de objetos.
El hash no debe incluir el propio contentHash.
El mismo contenido debe producir el mismo hash.
Dos proyectos distintos no deben compartir el mismo snapshot por accidente.

6. Payload

ProjectSnapshotPayload debe incluir únicamente el estado metodológico necesario para reconstruir el proyecto.

Como mínimo:

interface ProjectSnapshotPayload {
  problem: unknown;
  selectedThinkingPatterns: unknown[];
  reflections: unknown[];
  hypotheses: unknown[];
  conceptualModels: unknown[];
  operationalizationMatrices: unknown[];
  contrastationMatrices: unknown[];
  evidenceEvaluationMatrices: unknown[];
  hypothesisEvidenceConclusions: unknown[];
  reportDefinitions: unknown[];
}

Usar los tipos reales existentes, no unknown, en la implementación final.

Evaluar si deben incluirse también:

Knowledge Pack seleccionado;
decisiones del Router;
explicaciones;
selección de patrón principal y secundario.

Incluir únicamente lo necesario para reproducibilidad metodológica.

No incluir:

Knowledge Graph generado;
auditorías derivadas;
reportes generados;
validaciones;
resúmenes;
filtros;
estado visual.
7. Versión de esquema

Definir una constante:

CURRENT_PROJECT_SCHEMA_VERSION

Formato recomendado:

1.0.0

No confundir:

schemaVersion

con:

versionLabel

schemaVersion identifica la estructura técnica.

versionLabel identifica la versión metodológica declarada por el usuario.

8. Normalización determinística

Crear:

normalizeSnapshotPayload()

Debe:

ordenar claves de objetos;
preservar orden semántico de listas cuando sea relevante;
ordenar listas no semánticas de forma estable;
excluir campos transitorios;
normalizar valores undefined;
conservar null cuando sea significativo;
evitar referencias circulares;
producir una representación canónica.

Documentar qué listas mantienen orden del usuario, por ejemplo:

variables;
indicadores;
evidencias;
secciones de reporte.

No ordenar listas cuyo orden tenga significado metodológico o editorial.

9. Hash

Crear:

computeSnapshotHash()

Usar un algoritmo criptográfico disponible en el entorno, preferiblemente SHA-256.

El hash debe calcularse sobre:

schemaVersion
+
payload normalizado
+
metadata estable relevante

Excluir:

createdAt;
contentHash;
textos transitorios;
orden accidental.

Agregar:

verifySnapshotIntegrity()

Debe recalcular el hash y comparar.

No corregir un hash inválido automáticamente.

11. Inmutabilidad

Aplicar inmutabilidad real en tiempo de ejecución cuando sea razonable.

Opciones válidas:

Readonly;
Object.freeze;
deep freeze controlado;
copias defensivas.

Agregar pruebas que demuestren que modificar el estado activo después de crear el snapshot no cambia el snapshot.

16. Cambios

Crear:

type ProjectChangeType =
  | "added"
  | "removed"
  | "modified"
  | "reordered"
  | "unchanged";

Cada cambio debe incluir:

interface ProjectEntityChange {
  entityType: string;
  entityId: string;
  changeType: ProjectChangeType;

  before?: unknown;
  after?: unknown;

  changedFields?: string[];
  traceability?: {
    hypothesisId?: string;
    parentEntityId?: string;
  };
}

No generar descripciones con LLM.

Usar comparaciones determinísticas.

17. Detección de modificaciones

La comparación debe distinguir:

modificación de contenido

de:

reordenamiento

Ejemplo:

misma evidencia, distinta posición: reordered;
misma evidencia, descripción modificada: modified;
indicador eliminado: removed.

No usar comparación por índice únicamente.

Comparar entidades por ID.

18. Resumen de cambios

Crear:

interface ProjectDiffSummary {
  added: number;
  removed: number;
  modified: number;
  reordered: number;
  unchanged: number;
  affectedEntityTypes: string[];
  hasBreakingChanges: boolean;
}

hasBreakingChanges debe basarse en reglas explícitas.

Ejemplos de breaking change:

eliminación de hipótesis;
eliminación de variable referenciada;
eliminación de indicador usado;
eliminación de evidencia usada en conclusión;
referencia rota.

No considerar cualquier edición textual como breaking.

}

El hash del reporte debe calcularse sobre su representación canónica.

37. Eliminación de snapshots

No permitir eliminar un snapshot si:

una versión lo referencia;
otra versión depende de él;
es necesario para reconstruir una cadena.

Mostrar impacto antes de eliminar.

Permitir eliminación solo cuando no existan dependencias o tras eliminar primero las versiones relacionadas.
